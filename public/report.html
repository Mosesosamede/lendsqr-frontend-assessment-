Project Overview

This technical report details the architectural blueprint, data pipeline integrations, stylesheet systems, and critical engineering decisions made during the design and development of the Lendsqr Admin Dashboard. Built on Next.js, TypeScript, Supabase, and SCSS, this solution represents a production-ready system constructed according to strict visual guidelines and robust state management principles.

The Lendsqr Admin Dashboard is a mission-critical operations portal designed for management of borrowers, account parameters, credit facilities, and financial profiles. The primary objective is to provide administrators with an intuitive, dense, and interactive system for managing hundreds of thousands of users.

The implementation covers:

  • An enterprise-grade login screen featuring rigid validation, mock authorization routing, and pixel-perfect layouts.
  • A multi-metric dashboard featuring unified system analytics counters (Total Users, Active Users, Users with Loans, Users with Savings).
  • An advanced user directory table capable of paginating, filtering (across 6 dynamic criteria: Organization, Username, Email, Date, Phone Number, and Status), sorting, and performing context-aware administrative operations (Blacklist and Activate).
  • A deep user inspection module that maps comprehensive core records (Personal Profile, Education and Employment, Socials, and Guarantor Details) into structured tabbed views.

Project Architecture

The system follows a strict Full-Stack architecture where Next.js manages both user interface components and server-side API proxy routes. To secure database credentials and protect upstream services from unauthorized access, the browser client is restricted from directly communicating with Supabase. Instead, all data operations are routed through server-side Route Handlers.

Development Approach

The project implementation followed an organized development pipeline to ensure reliability, strict alignment with the design instructions, and zero regressions:

  1. Requirements Gathering and UI Spec Analysis: Hand-crafted mockups and brand guidelines were analyzed to determine the exact sizing ratios, padding configurations, color palettes, and typography metrics.
  2. Data Schema Design: The User interface structure was formalized in TypeScript. Fields were aligned with actual database tables to enable comprehensive querying.
  3. Sass Variable & Structural Architecture: Base configurations (such as primary blues, teal accents, fonts, spacing values, and screen mixins) were written into separate stylesheets before building individual pages. This prevented style overrides and layout misalignments.
  4. Next.js App Routing and Shell Assembly: Root layouts, authentication pages, dashboard wrappers, and detailed inspection views were created as client-side dynamic shells under the standard Next.js directory.
  5. Server API Routes and Database Provisioning: Relational query adapters were configured in route handlers, adding paginated endpoints, precise filter chains, and mathematical aggregations.
  6. Client Component Optimization and Interactivity: Active tables were equipped with dynamic filter popups, pagination dropdowns, hover states, and dynamic status actions. Caching strategies were introduced to allow sub-second page rendering.

Data Layer

The database layer uses PostgreSQL hosted on a scalable Supabase instance. It contains a comprehensive users table. Each user record includes over twenty fields to mirror real production data, including:

  • Core Profile: ID, full name, username, email, phone number, BVN, gender, marital status.
  • Educational/Employment Background: Education level, employment status, sector, monthly income, monthly loan repayment.
  • Financial Assets: Account balance, account number, credit score, current savings, bank code.
  • Metadata: Creation date, account status, guarantor names, guarantor contact info, and guarantor relationships.

To populate the database with realistic developer assessment records, a comprehensive seed migration script was executed during DB setup, populating the database with a high-density, real-world representative dataset.

API Design

To decouple client states from database structures, database requests are proxied through standard Next.js route handlers. This design ensures that credentials remain hidden, and query operations are optimized on the server before transferring JSON payloads to the browser.

1. User Database Endpoint (GET /api/users)

Retrieves user collections, supporting multi-criteria filtering, text searches, and exact relational pagination.

Query Parameters Supported:

  • page: Current segment index (default: 1).
  • limit: Elements per page (default: 10).
  • search: Full-text search string matched via ILIKE on names, emails, usernames, and telephone numbers.
  • organization: Hard filter mapping to exactly matched enterprise associations.
  • status: Exact status match (Active, Inactive, Pending, Blacklisted).
  • username / email / phone_number: Multi-input filters mapping to column segments.
  • date_joined: Date validation match tracking registration date bounds.

Example Server Query Construction:

let query = supabase.from('users').select('*', { count: 'exact' });

if (search) {
  query = query.or(`full_name.ilike.%${search}%,email.ilike.%${search}%,phone_number.ilike.%${search}%,username.ilike.%${search}%`);
}
if (status) {
  query = query.eq('status', status);
}
// Paginate
query = query.range(from, to);

2. Metrics Summary Endpoint (GET /api/stats)

Executes database queries to calculate aggregate totals. This endpoint avoids loading entire records into memory by requesting database counts directly:

const { count: totalCount } = await supabase.from('users').select('id', { count: 'exact', head: true });
const { count: activeCount } = await supabase.from('users').select('id', { count: 'exact', head: true }).eq('status', 'Active');
const { count: loanCount } = await supabase.from('users').select('id', { count: 'exact', head: true }).gt('loan_repayment', 0);
const { count: savingsCount } = await supabase.from('users').select('id', { count: 'exact', head: true }).gt('account_balance', 0);

Frontend Architecture

The client interface is built using standard React components, TypeScript interfaces, and maintainable SCSS layout definitions.

1. Styling and SCSS Organization

Styles are modular and completely decoupled from React structures, avoiding inline styles or heavy utility class overrides. The styling system is compiled from an organized directory:

  • _variables.scss: Defines the official colors (Primary Deep Blue: #213F7D, Secondary Accent Teal: #39CDCC, Text Gray: #545F7D), font weights, and border radius targets.
  • _mixins.scss: Contains responsive viewport media-queries (e.g., mobile, tablet, desktop, large-desktop) to keep layouts responsive across different screens.
  • components/: Separates styling blocks for _table.scss, _stat-cards.scss, and _filters.scss.
  • pages/: Includes specific configurations for login layout structures, details sections, and dashboard grids.

2. Form Architecture

Forms are managed through react-hook-form and validated via @hookform/resolvers/zod. Form rules are declared as strict Zod schemas, separating error tracking from UI rendering logic:

export const loginSchema = z.object({
  email: z.string().min(1, 'Email is required').email('Enter a valid email address'),
  password: z.string().min(6, 'Password must be at least 6 characters'),
});

State Management

The application employs a hybrid state strategy, choosing the most appropriate tool for each use case:

  • Server-Client Sync: Local filter parameters are stored inside React state within parent containers. Updating pagination indexes or filter fields triggers an asynchronous data fetch, updating users and stats components seamlessly.
  • LocalStorage Caching: Dynamic inspection views (UserDetailPage.tsx) query cached representations first before making API calls. This enables sub-second load times when clicking through user directories.
  • State Flags: Comprehensive loading indicators, empty lists, and error feedback elements are configured as react states, preventing page freezing or broken displays on network lag.

Performance Considerations

Several optimizations are implemented in the codebase to ensure responsive performance under high data loads:

1. Relational Database Range Offsets

The server-side endpoint processes pagination using range boundaries (.range(from, to)) rather than loading complete data slices. This reduces transfer sizes and network overhead.

2. Progressive Loading & Non-Blocking Caching

When an administrator opens a dynamic user inspection page, the application uses a progressive loading strategy:

  1. The client checks localStorage for cached user data (keyed by user ID).
  2. If found, the cached data is loaded immediately, providing an instant render.
  3. In the background, a non-blocking fetch call is sent to the server.
  4. When the server responds with fresh data, the UI is updated, and the cache is refreshed.
// 1. Check localStorage first for immediate UI
const cachedData = localStorage.getItem(STORAGE_KEY);
if (cachedData) {
  setUser(JSON.parse(cachedData));
  setLoading(false); 
}

// 2. Fetch fresh data from API in background
const freshUser = await api.getUserById(id);
setUser(freshUser);

// 3. Update cache
localStorage.setItem(STORAGE_KEY, JSON.stringify(freshUser));

Accessibility

The user interface complies with accessible web practices to ensure usability for all administrators:

  • Semantic Elements: Core page layouts use distinct structural elements (<header>, <aside>, <main>, <section>) instead of generic division containers.
  • Form Elements: Standard validation fields map explicit identifiers to form tags using labels (<label htmlFor="username">).
  • ARIA Elements: Dynamic field validations communicate error structures using appropriate attribute settings (aria-invalid="true").
  • Visual Contrast: Foreground elements are styled to maintain high visual contrast relative to their backgrounds, ensuring readability.

Responsive Design

The design adapts fluidly across mobile, tablet, and wide desktop viewports:

  • Desktop Screens: The layout features a dense structural layout, with a persistent left-hand navigation sidebar, search bars, global system header controls, and a multi-column data grid.
  • Tablet viewports: Responsive Sass mixins convert the main grid into a single-column block. The navigation sidebar transitions into an absolute-positioned slide-out panel, accessible via a mobile menu trigger.
  • Mobile devices: Input filters shift into a compact drawer format to fit smaller screens, ensuring form inputs remain legible. Tables are wrapped in horizontally scrollable containers to handle multi-column layouts without content clipping.

Engineering Decisions & Tradeoffs

During the planning and development phases, several architectural tradeoffs were evaluated:

1. Server-Side Routing vs. Single-Page (SPA) Layouts

Decision: Next.js App Router.

Alternative Considered: Create React App (SPA).

Tradeoff: While an SPA is slightly simpler to set up initially, Next.js provides optimized route splitting, server-side API proxying, and built-in route compilation, resulting in faster load times and better security boundaries.

2. Supabase Integration vs. In-Memory Mock Arrays

Decision: Supabase PostgreSQL.

Alternative Considered: In-memory mock arrays.

Tradeoff: Implementing a real cloud database requires more initial setup and schema design. However, it is the only way to support real pagination, sorting, and multi-criteria filtering across large datasets without overloading browser memory.

3. SCSS Variables vs. Utility CSS Frameworks

Decision: Custom Sass/SCSS Modular System.

Alternative Considered: CSS-in-JS or inline utility styles.

Tradeoff: Inline styles and CSS-in-JS can increase JavaScript bundle sizes. By choosing a precompiled SCSS architecture, we keep stylesheets separated from component logic, resulting in faster styling lookups and more maintainable stylesheets.

Challenges & Solutions

During implementation, several technical challenges were identified and resolved:

1. Optimizing Bundle Sizes and High-Memory Builds

Problem: During production compilation, the bundler encountered out-of-memory errors on container environments.

Solution: The Next.js webpack script was configured with custom node options (NODE_OPTIONS=--max-old-space-size=1024) to restrict maximum memory consumption. In addition, unneeded dependencies were pruned, reducing bundle overhead and build times.

2. Table Layout and Multi-Criteria Filtering

Problem: Managing the layout of the user table while supporting six distinct filtering criteria created complex form state combinations.

Solution: A dedicated, reusable filtering component (FilterPopup.tsx) was created. The component mounts directly inside the table headers and reads its initial states from active parent filters. This keeps the layout organized and maintains clean state transitions.

Testing Strategy

The application features automated testing pipelines executed via Vitest. The testing suite focuses on validating core schemas, form inputs, and validation criteria:

  • Core Validations: Tests confirm that proper email validation rules are enforced, rejecting poorly formatted entries and displaying appropriate warnings.
  • Password Constraints: Asserts that passwords under six characters are rejected, protecting authentication entry points.
  • State Validation: Confirms that correct credentials validate successfully, ensuring reliable routing.

Security Considerations

Security is built into the application at the structural level:

  • Hidden Secrets: Database connection keys (SUPABASE_URL and SUPABASE_ANON_KEY) are stored as server-side environment variables and are never exposed to the client.
  • Validation Boundaries: Form inputs are validated on the server via Next.js route handlers, preventing unauthorized queries from reaching the database.
  • Input Cleansing: Form inputs are sanitized before being executed as database queries, protecting database operations.

Future Improvements

The following enhancements are proposed for future development phases:

  1. SSO Authentication: Integrate OAuth flows (Google Workspace, Microsoft Entra ID) to replace credential-based authentication with secure, single-sign-on access.
  2. Export Capabilities: Add CSV and Excel export utilities directly on the table wrapper, allowing administrators to export filtered and paginated datasets.
  3. Multi-Column Sorting: Support sorting across multiple table columns simultaneously, enabling more complex data analysis.
  4. Comprehensive Audit Logging: Track all administrative actions (such as activating or blacklisting users) in a secure audit table to log operational actions.

Conclusion

The Lendsqr Admin Dashboard represents a highly polished, responsive, and robust solution. By implementing a Next.js full-stack architecture with Supabase integrations, modular SCSS stylesheets, and local-first progressive caching, the application delivers reliable performance and meets all visual and functional guidelines. The system is fully compiled, tested, and ready for deployment on Vercel.
